# AI 提交规划器使用指南

Gitora v1.3.0 可以连接用户已有的本地 Ollama 服务或远程 Responses API，生成提交信息，并把一组工作区改动规划为多个文件级或代码块级提交。

核心边界始终不变：模型只提出建议，Gitora 校验并准备每一组改动，是否提交和何时推送仍由用户决定。该功能不会自动连续提交，不会自动推送，也不会让模型运行 Git、Shell 或项目命令。

## 启用与配置

打开“设置 → AI 提交规划”，开启功能后选择一种提供方。功能默认关闭，端点和模型名默认留空；Gitora 不内置模型权重，也不替用户选择服务地址。

### 本地 Ollama

填写：

- Ollama 服务地址，例如 `http://<ollama-host>:<port>`；地址可使用 HTTP 或 HTTPS，但不能包含用户名、密码、查询参数或 URL 片段。
- 已在该服务中安装的模型名。

“检测连接”会读取服务提供的模型列表并确认所填模型存在。实际生成时，源码上下文只发往该配置地址。若该地址指向局域网或公网主机，数据仍会离开当前电脑，请按实际部署边界判断隐私风险。

### 远程 Responses API

填写：

- 完整的 HTTPS Responses API 地址；不接受 HTTP、URL 内凭据、查询参数或片段。
- 远程模型名。
- 保存密钥的环境变量名，或本次运行使用的会话密钥。

推荐把密钥放在环境变量中，并只在设置里填写变量名。环境变量名只允许字母、数字和下划线，且不能以数字开头。会话密钥只保存在当前进程内，退出 Gitora 后清除；保存设置时不会把密钥写入 `ai_commit.json`。

每次真正发送源码差异前，Gitora 都会显示分析范围、变更数量和预计字符数。只有用户点击“确认发送”后才会调用远程服务；取消确认不会上传。

### 规划范围

范围设置只控制“规划提交”，可选：

- 仅已暂存差异：文件级和代码块级规划都只分析索引中的改动。
- 全部工作区改动：规划会分析已暂存、未暂存和未跟踪改动。

“AI 生成”单条提交信息始终只读取已暂存差异，不受该范围设置影响。

“生成提交正文”关闭后，模型仍生成标题，但正文必须为空。

## 使用流程

### 生成一条提交信息

1. 在仓库页暂存准备提交的改动。
2. 点击提交输入框右侧的“AI 生成”。
3. 使用远程提供方时，检查发送摘要并明确确认；本地提供方会直接调用已配置的本地服务。
4. 生成结果会填入现有提交输入框。先人工审阅和编辑，再点击“提交”。

生成期间可以点击“取消生成”。生成行为不会暂存、取消暂存、提交或推送任何内容。

### 文件级提交规划

1. 点击“规划提交”，选择“文件级规划”。
2. 查看模型建议的提交组、标题、正文、拆分理由、依赖和警告。
3. 编辑提交信息，移动文件或调整组顺序，并查看每组实际补丁。
4. 点击“应用下一组”。Gitora 会先复验工作区和计划覆盖，再只准备当前组。
5. 回到仓库页复核暂存差异，使用现有“提交”按钮提交。
6. Gitora 核验实际提交树后才会推进到下一组；重复以上步骤直到完成。

### 代码块级提交规划

代码块级流程允许同一文件的不同 hunk 进入不同提交组。规划对话框会显示每个代码块的上下文、暂存状态、所属提交和覆盖率；用户可以手动移动代码块并预览当前组的实际补丁。

执行前，Gitora 会校验每个代码块恰好出现一次，并在隔离索引中试应用整份计划。实际应用仍然一次只处理一组；提交成功后，剩余代码块必须能够与新的工作区快照稳定重映射，否则计划会失效并要求重新生成。

## 发送给模型的数据

一次规划请求可能包含：

- 当前 `HEAD`、分支名和是否包含未暂存改动。
- 选定范围内的相对路径、差异正文或代码块上下文、变更类型和增删统计。
- 最近的提交标题，用于推断仓库的语言和标题风格。
- 仓库根目录中允许读取的规范文件；v1.3.0 默认仅为 `AGENTS.md` 和 `CONTRIBUTING.md`，并受字符上限约束。
- 快照标识、稳定变更标识、完整性警告和结构化输出要求。

请求不包含仓库绝对路径。Gitora 不会把 API 密钥写入提示词，也不会让模型访问请求范围外文件。日志只记录错误类型和状态，不记录授权请求头或发送的源码正文。

二进制、子模块、未解决冲突、符号链接和超过上限的内容会被标记为不支持或不完整，不能绕过确定性校验直接执行。

## 安全与恢复

- 模型返回内容被视为不可信数据，必须通过协议、标识覆盖、工作区指纹和补丁试应用校验。
- 计划生成后，只要 `HEAD`、索引、工作区或未跟踪文件发生变化，执行前复验就会拦截过期计划。
- 每次只应用一组；上一组未由用户提交前，不能继续下一组或清空当前计划。
- 应用或提交核验失败后停止推进，不会静默跳过失败组，也不会重写历史。
- 代码块计划使用隔离索引预验；真实索引写入失败时会尝试恢复原索引状态。
- 功能不会自动推送。任何推送仍走 Gitora 现有的显式操作和危险确认。

开始规划前仍建议保存无法重建的重要工作。Gitora 的恢复保护不能替代备份。

## 故障排查

### 提示“请先在设置中启用 AI 提交规划”

打开设置中的功能开关并保存。

### 提示服务地址或模型名无效

- 本地地址必须是有效 HTTP(S) URL，模型名必须已存在于所连接的 Ollama 服务。
- 远程地址必须是完整 HTTPS Responses API 地址，模型名不能为空。
- 点击“检测连接”获取具体的网络、HTTP、模型列表或响应格式错误。

### 提示“未提供远程模型密钥”

确认环境变量已在启动 Gitora 的同一进程环境中设置，且设置页填写的是变量名而不是密钥值；也可以填写只保留到退出的会话密钥。

### 暂存区为空或没有可规划改动

- “AI 生成”要求至少一项已暂存改动。
- 规划范围为“仅已暂存差异”时，也必须先暂存改动。
- 若要分析未暂存或未跟踪改动，把规划范围改为“全部工作区改动”。

### 计划已过期或工作区已变化

规划后发生了新的编辑、暂存、取消暂存、提交、分支切换或仓库切换。关闭旧计划并基于当前状态重新生成。

### 模型响应无效、覆盖不完整或补丁无法应用

模型输出没有满足 Gitora 的结构化协议，遗漏/重复了变更，或者建议无法在隔离索引中安全应用。此类计划不会进入执行；可以重新生成、缩小改动范围或手动提交。

### 请求超时、响应过大或远程限流

检查服务状态和网络，缩小规划范围后重试。网络超时、输入上限和响应上限来自 Gitora 配置；v1.3.0 的图形界面只开放常用提供方字段，不开放这些高级限制。

### 二进制、子模块、冲突或大型差异被拒绝

这是安全降级而非模型故障。先解决冲突，拆小差异，或使用 Gitora 的普通 Git 流程手动处理不支持的变更。

## 已知限制

- 安装包不包含本地模型；模型安装、显存/内存需求和服务维护由用户负责。
- 首版只直接支持 Ollama 和兼容预期响应结构的 Responses API。
- 模型建议可能需要改标题、合并或重新分组；模型质量不代表代码正确或测试通过。
- Gitora 不会自动运行项目测试，提交前仍需由用户执行项目自己的验证命令。
- 超大或不完整快照会被拒绝，不能通过静默截断换取一个看似可用的计划。
- 代码块拆分以 Git 统一差异 hunk 为边界，不是语法树级重构。
- 当前密钥方式为环境变量或进程内会话值，尚未接入系统凭据保险库。
- v1.3.0 已建立 30 组真实历史回放集和本地/远程人工评价表；在未配置并实际运行模型前，不宣称任何模型达到质量门槛。

## 关闭功能

在“设置 → AI 提交规划”关闭开关并保存即可。已有普通暂存、提交和推送流程不依赖 AI 配置。

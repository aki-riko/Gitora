# AI 提交规划器使用指南

Gitora v1.3.0 可以连接用户已有的本地 Ollama 服务、远程 OpenAI 兼容 API（Chat Completions / Responses）或 Anthropic Messages API，生成提交信息，并把一组工作区改动规划为多个文件级或代码块级提交。

核心边界始终不变：模型只提出建议，Gitora 校验每一组改动；用户点击“采用并提交推送”后，Gitora 才会按组提交，并在所有组成功后一次推送。提交标题、正文、summary、rationale 和 warnings 会跟随当前 UI 语言生成；该功能不会让模型运行 Git、Shell 或项目命令。

## 启用与配置

打开“设置 → AI 提交规划”，开启功能后选择一种提供方。功能默认关闭，端点和模型名默认留空；Gitora 不内置模型权重，也不替用户选择服务地址。

### 本地 Ollama

填写：

- Ollama 服务地址，例如 `http://<ollama-host>:<port>`；地址可使用 HTTP 或 HTTPS，但不能包含用户名、密码、查询参数或 URL 片段。
- 已在该服务中安装的模型名。

“检测连接”会读取服务提供的模型列表并确认所填模型存在。实际生成时，源码上下文只发往该配置地址。只有 `localhost`、`127.0.0.0/8` 或 `::1` 等本机回环地址会直接发送；若地址指向局域网或公网主机，数据会离开当前电脑，Gitora 会在每次发送前显示范围并要求明确确认。

### 远程 OpenAI 兼容 API / Anthropic Messages API

填写：

- OpenAI 兼容 API 填写 HTTPS 基础地址或完整的 Chat Completions / Responses 地址；Anthropic 填写 `https://api.anthropic.com`、`/v1` 基础地址或完整的 `/v1/messages` 地址。不接受 HTTP、URL 内凭据、查询参数或片段。OpenAI 地址以 `/responses` 结尾时使用 Responses 协议，其余地址按 Chat Completions 协议自动补全；Anthropic 地址按 Messages 协议补全。
- 远程模型名。
- API 密钥。图形界面保存时会写入当前用户的系统凭据库。
- 可选的密钥环境变量名，供自动化启动或高级兼容场景使用。

Windows 使用 Credential Manager，macOS 使用 Keychain；密钥由操作系统按当前用户保护，Gitora 不创建自有明文或 AES 密钥文件。保存成功后密钥会跨 Gitora 重启保留，设置页可以显式删除。密钥按提供方和精确端点隔离，切换远程端点不会复用旧端点的密钥。

`ai_commit.json` 始终只保存非敏感配置。若同时存在系统凭据和环境变量密钥，系统凭据优先。环境变量名只允许字母、数字和下划线，且不能以数字开头。

点击提交菜单中的“AI 提交”即开始本次规划；远程模型请求会发送当前仓库的已暂存、未暂存和未跟踪改动。生成结果框只负责让用户审阅计划，点击“不采用”不会产生 Git 写操作。

### 规划范围

AI 提交固定分析全部工作区改动，包括已暂存、未暂存和未跟踪文件。旧版配置中的 `staged` 范围会自动迁移为 `all`，无需手动修改配置文件。

“生成提交正文”关闭后，模型仍生成标题，但正文必须为空。

## 使用流程

### 生成并采用分组提交计划

1. 在仓库页准备需要提交的改动。
2. 点击提交下拉菜单中的“AI 提交”。
3. 等待进度框完成，结果框会显示 AI 规划出的 Commit 数量、每组标题、正文和改动数量。
4. 点击“不采用”不会产生 Git 写操作；点击“采用并提交推送”后，Gitora 会按显示顺序逐组提交，全部组成功后一次推送。

生成期间可以点击“取消生成”。生成阶段不会暂存、取消暂存、提交或推送任何内容。

### 文件级提交规划

模型建议的每个提交组都会先经过协议、覆盖范围、工作区指纹和索引隔离校验。确认后，Gitora 自动完成逐组应用与提交；任一组失败都会停止后续组，不会静默合并成一个大 Commit。

文件级计划若包含已暂存的新增、重命名、复制或文件模式变更，必须把这些依赖索引元数据的改动集中到第一提交组；否则 Gitora 会把计划标记为不可执行，避免它们在移出暂存区时退化或丢失。可在计划界面调整分组与顺序后重新应用。

### 代码块级提交规划

代码块级流程允许同一文件的不同 hunk 进入不同提交组。规划对话框会显示每个代码块的上下文、暂存状态、所属提交和覆盖率；用户可以手动移动代码块并预览当前组的实际补丁。

执行前，Gitora 会校验每个代码块恰好出现一次，并在隔离索引中试应用整份计划。实际应用仍然一次只处理一组；提交成功后，剩余代码块必须能够与新的工作区快照稳定重映射，否则计划会失效并要求重新生成。

## 发送给模型的数据

一次规划请求可能包含：

- 当前 `HEAD`、分支名和是否包含未暂存改动。
- 选定范围内的相对路径、差异正文或代码块上下文、变更类型和增删统计。
- 最近的提交标题，用于推断仓库的语言和标题风格。
- 仓库根目录中允许读取的规范文件；v1.3.0 默认仅为 `AGENTS.md` 和 `CONTRIBUTING.md`，并受字符上限约束。
- 快照标识、稳定变更标识、完整性警告和结构化输出要求。

请求不包含仓库绝对路径。Gitora 不会把 API 密钥写入提示词，也不会让模型访问请求范围外文件。日志只记录错误类型和状态，不记录授权请求头或发送的源码正文。

二进制、子模块、未解决冲突、符号链接和超过上限的内容会被标记为不支持或不完整，不能绕过确定性校验直接执行。

## 安全与恢复

- 模型返回内容被视为不可信数据。所有计划都必须通过协议、标识覆盖和工作区指纹校验；代码块级计划还必须通过隔离索引补丁试应用。
- 计划生成后，只要 `HEAD`、索引、工作区或未跟踪文件发生变化，执行前复验就会拦截过期计划。
- 每次只应用一组；上一组提交并核验成功后才会继续下一组。
- 应用或提交核验失败后停止推进，不会静默跳过失败组，也不会重写历史。
- 文件级计划会先保存原索引树再写入当前组，失败时尝试恢复；代码块级计划还会先使用隔离索引预验整份计划。
- 只有用户在结果框点击“采用并提交推送”才会触发最终推送；推送只发生一次，并且位于全部 Commit 成功之后。

开始规划前仍建议保存无法重建的重要工作。Gitora 的恢复保护不能替代备份。

## 故障排查

### 提示“请先在设置中启用 AI 提交规划”

打开设置中的功能开关并保存。

### 提示服务地址或模型名无效

- 本地地址必须是有效 HTTP(S) URL，模型名必须已存在于所连接的 Ollama 服务。
- 远程地址必须是 HTTPS API 基础地址或完整的 Chat Completions / Responses / Messages 地址，模型名不能为空。
- 点击“检测连接”获取具体的网络、HTTP、模型列表或响应格式错误。

### 提示“未提供远程模型密钥”

在设置页填写 API 密钥并保存，确认状态显示“已保存到系统凭据库”。也可以确认环境变量已在启动 Gitora 的同一进程环境中设置，且“密钥环境变量名”填写的是变量名而不是密钥值。若系统凭据库不可用，设置页会显示操作系统后端的脱敏错误，不会降级为明文文件。

### 没有可规划改动

AI 提交会同时检查已暂存、未暂存和未跟踪改动；三者均为空时才会提示没有可规划内容。

### 计划已过期或工作区已变化

规划后发生了新的编辑、暂存、取消暂存、提交、分支切换或仓库切换。关闭旧计划并基于当前状态重新生成。

### 模型响应无效、覆盖不完整或补丁无法应用

模型输出没有满足 Gitora 的结构化协议，遗漏/重复了变更，或者建议无法在隔离索引中安全应用。此类计划不会进入执行；可以重新生成、缩小改动范围或手动提交。

### 请求超时、响应过大或远程限流

检查服务状态和网络，缩小规划范围后重试。网络超时、输入上限和响应上限来自 Gitora 配置；v1.3.0 的图形界面只开放常用提供方字段，不开放这些高级限制。

### 二进制、子模块、冲突或大型差异被拒绝

这是安全降级而非模型故障。先解决冲突，拆小差异，或使用 Gitora 的普通 Git 流程手动处理不支持的变更。

## 已知限制

- 安装包不包含本地模型；模型安装、显存/内存需求和服务维护由用户负责。
- 首版直接支持 Ollama、兼容预期响应结构的 OpenAI Chat Completions / Responses API，以及 Anthropic Messages API。
- 模型建议可能需要改标题、合并或重新分组；模型质量不代表代码正确或测试通过。
- Gitora 不会自动运行项目测试，提交前仍需由用户执行项目自己的验证命令。
- 超大或不完整快照会被拒绝，不能通过静默截断换取一个看似可用的计划。
- 代码块拆分以 Git 统一差异 hunk 为边界，不是语法树级重构。
- 原生凭据库当前支持 Windows Credential Manager 与 macOS Keychain；其他操作系统只能使用环境变量兼容路径。
- v1.3.0 已建立 30 组真实历史回放集和本地/远程人工评价表；在未配置并实际运行模型前，不宣称任何模型达到质量门槛。

## 关闭功能

在“设置 → AI 提交规划”关闭开关并保存即可。已有普通暂存、提交和推送流程不依赖 AI 配置。
